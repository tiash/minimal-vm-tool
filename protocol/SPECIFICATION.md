# Minimal VM Exec Protocol Specification

## Overview

The Minimal VM Exec Protocol is a JSON-based communication protocol implemented over virtio-vsock. It enables the host to spawn and control processes inside the VM.

## Protocol Port

- **Port Name:** `"exec"`
- **Port Number:** `1702198635` (big-endian encoding of `"exec"` as u32)

## Connection Lifecycle

1. HOST initiates connection to VM on the exec port
2. HOST sends `exec` message to spawn a process
3. VM responds with `started` message upon successful process spawn
4. Bidirectional communication continues for stdin/stdout/stderr
5. Process exits and VM sends `exited` message
6. VM continues stdout/stderr communication until both have been closed
7. HOST closes connection (after receiving `exited`, `stdout.closed` and `stdin.closed`)

### Stdout/Stderr/Stdin Lifecycle

For stdout/stderr VM is SOURCE and HOST is SINK.
For stdin HOST is SOURCE and VM is SINK.

General flow is
1. SOURCE sends `data` messages for every line or chunk depending on the `line_bufferd_io` parameter on exec. (When sending line-based data, each line's CONTENT must include the trailing '\n').

2. when the stream is closed and all data has been sent the SOURCE must send a `closed` message.

At any time the SINK may send a `close` message to indicate that the SOURCE should stop sending further `data` messages.
The SOURCE may continue to send any buffered data (but the SOURCE should prevent further buffering of new data, e.g. the VM should close the read end of processes stdout/stderr) and must send a `closed` to acknowledge once all remaining buffered data is sent (as per the general flow above).
The SINK may silently discard any `data` messages received after it sent a `close`.

## Message Format

All messages are JSON objects serialized as UTF-8 strings, terminated by newline (`\n`).

The general message structure is
```json
{ "VERB": ... }
```
where `VERB` is one of `exec`, `started`, `stdio`, `stdout`, `stderr`, `kill`, `exited` or `error` as described below.

### exec (HOST -> VM)
Spawn a new process in the VM.

```json
{ "exec":
  { "prog": "string",
    "args": [ "arg1", "arg2", ... ],
    "pwd": "/path/to/working/directory",
    "user": "optional_username",
    "line_buffered_io": boolean
  }
}
```

**Fields:**
- `prog`: absolute path of the executable to run
- `args` (optional): array of command-line arguments (default `[]`)
- `pwd` (optional): working directory for the process (optional, default `"/"`)
- `user` (optional): Username to run as (Unix only), (default `"user"`)
- `line_buffered_io` (optional): Whether IO should be line-buffered or chunked (default `false`)

**Response:**
VM sends `started` message on success, or `error` on failure.

**Behaviour:**
This must be the first message the host sends.
Any other message is considered an `error`.

---

### started (VM -> HOST)
The process has been successfully spawned.

```json
{ "started": { } }
```

**Fields**
Unspecified, VM may include any appropriate metadata about the process.

**Response:** 
None

**Behaviour:**
This or `error` must be the first message sent by VM.

---

### stdin/stdout/stderr

These are messages describing the input and output of the process described in more detail below.

The general structure is 

```json
{ "stdin": { "SUB_VERB": ... } }
{ "stdout": { "SUB_VERB": ... } }
{ "stderr": { "SUB_VERB": ... } }
```
where `SUB_VERB` is one of `data`, `close` or `closed` as described below.

#### stdin.data (HOST -> VM) stdout.data, stderr.data (VM -> HOST)
input or output data transmission (line or chunk)

```json
{ "stdin": { "data": "CONTENT" } }
{ "stdout": { "data": "CONTENT" } }
{ "stderr": { "data": "CONTENT" } }
```

**Fields:**
- `data`: String content chunk/line (including trailing `\n`) of the stream

**Note:** Multiple data messages may be sent. They should be processed in order.

**Response:** None

**Behaviour:**
the SOURCE (VM for `stdout`/`stderr`, HOST for `stdin`) will send this repeatedly as output is generated by the process (VM) / input should be sent to the process (HOST).

Once all input/output has been sent SOURCE must send a `closed` message.

---

#### stdin.closed (HOST -> VM), stdout.closed, stderr.closed (VM -> HOST)
Indicate the end input or output.

```json
{ "stdin": { "closed": true } }
{ "stdout": { "closed": true } }
{ "stderr": { "closed": true } }
```

**Fields:**
- `closed`: the stream is closed, always `true`.

**Response:** None

**Note:**
Once `closed` has been sent no further `data` may be sent.

**Behaviour:**
For stdin signal the VM process that all input has been received.
For stdout/stderr signal to the HOST that no further output will be produced.

If a `data` message is received and the underlying destination has already been closed it should be ignored. However it is considered an error to send a `data` message after the corresponding `closed` message.

---

#### stdin.close (VM -> HOST), stdout.close, stderr.close (HOST -> VM)
Indicate that no further input/output will be consumed.

```json
{ "stdin": { "close": true } }
{ "stdout": { "close": true } }
{ "stderr": { "close": true } }
```

**Behaviour:**
VM will signal the process to stop producing output on stdout/stderr as appropriate.
HOST should stop sending input to stdin as appropriate.

Any further `data` messages for that stream may be ignored.

The remote may continue to send `data` messages until buffers are flushed.
Once all data is flushed a `closed` message must follow.

If a `close` message is received after `closed` was sent it should be ignored.

---

### kill (HOST -> VM)
VM terminates the process immediately with SIGKILL (kill -9).

```json
{ "kill": { } }
```

**Response:**
- VM sends `exited` message with exit status (once process is terminated, as per normal flow)
- VM sends any remaining buffered `stdout`/`stderr` data (as per normal flow)
- VM sends `stdout.closed` and `stderr.closed` (if these were not sent already, as per normal flow)

**Behavior:** 
- The VM will cause the running process to terminate immediately (unix kill -9)
- The connection remains open to allow the buffers to be flushed and ensure the host can receive all responses.

---

### exited (VM -> HOST)
Sent when process exits (after all buffered output has been sent).

```json
{ "exited": { "status": EXIT_CODE } }
```

**Fields:**
- `status`: Exit status code (integer)

**Response:** None

**Behaviour:**
Sent once the process exits.
The VM should continue to send any remaining buffered `stdout`/`stderr` data.
The VM must send the final `stdout.closed`/`stderr.closed` (if not already sent).

---

### error (VM -> HOST)
Sent when protocol violation or error occurs.

```json
{ "error": { "message": "MESSAGE" } }
```

**Fields:**
- `message`: Human readable description of the error

Optionally the VM may include other fields with additional detail.

**When to send:**
- Protocol violation (unexpected message in sequence)
- Failed to perform requested action
- Parse errors
- Any other error condition

**Behavior after error:**
- VM must kill the running process
- VM should ignore all further messages on the connection.
- HOST must close the connection.
- VM may close the connection.

---

### Connection Termination

**Normal process termination:**
1. Process exits
2. VM sends `exited` with status
3. VM sends `stdout.closed` and `stderr.closed`
4. HOST closes connection

**Error termination:**
1. VM sends `error` message
2. VM kills all processes
3. Connection may be closed

**Force termination (connection close):**
1. Any side closes/closed connection
2. VM kills processes immediately
3. All buffered data discarded

---

### Error Conditions (on VM side)
**Protocol violations:**
- Unexpected message type in current state, e.g.
  - `stdin.data` after `stdin.closed` or `kill`.
  - first message from HOST is not `exec` 
  - `exec` as any message other than first from HOST
- Missing required fields
- Invalid JSON format or json parsing errors.
**Action Failures:**
- Process spawn failure
- Process already killed
**Connection Errors:**
- Underlying socket is closed (either read or write side), unless process has exited and all closed messages sent.

**VM Behavior on Error:**
- VM should send an `error` message to the HOST (if the connection remains open) with a description.
- VM should send no further messages after the error.
- VM must kill the process (if started).
- VM must ignore any further messages from HOST.
- VM may close the connection after some timeout (30s)
- HOST must close the connection after receiving the `error` message

**HOST Behaviour on Error:**
- HOST must close the connection

## Implementation Notes

### Line Delimiting
Messages are delimited by newline (`\n`). Implementations must:
- Send exactly one `\n` after each JSON message
- Handle messages that may arrive concatenated or split across TCP packets

### Buffering
- VM should buffer stdin data until process is ready
- VM should flush stdout/stderr as available
- On close signals, drain all buffers before sending `closed` message, there is no timeout. If the underlying transport has been closed and we are unable to deliver the data it should be discarded.

### Concurrency
- Multiple processes may run concurrently on different connections.

### Line Buffering
When `line_buffered_io` is true:
- stdout/stderr should be flushed on newlines
- May affect timing of data delivery

---

## Security Considerations

1. **Authentication:** Connection should be restricted to host-only via virtio-vsock
2. **Input Validation:** VM should validate all message fields
3. **Resource Limits:** VM should enforce process/IO limits to prevent DoS
4. **User Context:** Process runs with permissions of `user` (or whichever was specified in the `exec` message) within the scope of the VM.

---

## Example Exchange

```text
HOST → VM: { "exec": { "prog": "echo", "args": [ "hello" ] } }
VM → HOST: { "started": { } }
VM → HOST: { "stdout": { "data": "hello\n" } }
VM → HOST: { "stdout": { "closed": true } }
VM → HOST: { "exited": { "status": 0 } }
HOST → VM: [close connection]
```
